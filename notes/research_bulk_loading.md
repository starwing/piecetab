# B+ 树 Bulk Loading 算法

> 来源: Wikipedia "B+ tree" 条目 § Bulk-loading (2026-05-28 获取)

## 算法本体

给定一批数据记录，欲为其建立 B+ 树索引。逐条插入 (insert each record into empty tree) 代价高昂——每次须从根向下遍历至叶。**Bulk loading** 是高效替代。

### 步骤

1. **排序**：将数据按搜索键升序排列
2. **建根**：分配空页为根，插入指向首条记录页的指针
3. **根满则裂**：根满时，分裂根并创建新根页
4. **逐项填入最右索引页**：持续向叶层紧上方的最右索引页插入条目，直至所有条目皆已索引

### 关键特性

- 当叶层上方的最右索引页填满时，该页分裂
- 此分裂可能引发更靠近根的最右索引页连续分裂
- **分裂仅发生于从根到叶的最右路径 (right-most path from the root to the leaf level)**

## 算法分析

### 与逐条插入之对比

| 特性 | 逐条插入 (one-by-one) | Bulk Loading |
|------|----------------------|-------------|
| 时间复杂度 | O(n log n) | O(n) |
| 分裂位置 | 任意路径 | 仅最右路径 |
| 叶填充率 | ~50% (裂叶均分) | ~100% (叶满方裂) |
| 适用场景 | 增量插入 | 大数据初始加载 |

### 填充率之优

逐条插入中，裂叶取中点将原叶一分为二（各 ~50%），左半永不再填，叶填充率仅 ~50%。Bulk loading 则叶满方裂，新数据填入新叶而不扰旧叶，叶填充率近 100%。

### 右路径分裂之机制

bulk loading 之所以高效，端赖"最右路径"特性：
- 新数据始终追加于树尾，故仅最右叶可能满
- 最右叶满 → 裂为二 → 新叶指针插入最右内节点
- 最右内节点若满 → 裂为二 → 新内节点指针上推至父
- 此过程仅沿最右路径上溯，不涉树中其他节点

此与 piecetab 之 `pend[]` 级联算法本质相通——皆自底向上建树，避逐条根遍历。

## 与 piecetab pend 算法之对应

| Bulk Loading 概念 | piecetab pend 对应 |
|-------------------|-------------------|
| 排序输入 | `S->linesf` 回调逐段扫描输入 (已排好序) |
| 填最右叶 | `ptI_fill` → `pend[levels]` 填满一叶 (62 Piece) |
| 叶满裂叶 | `ptI_flush` → `ptI_splitinsert` 将溢出推入上层 |
| 内节点满裂 | 同上，自底向上递归上推 |
| 根满裂根 | `ptI_splitroot` |
| 仅最右路径 | pend 数组级联即沿最右路径上推 |

## linecache 当前 lc_scan 之偏差

| 对比点 | Bulk Loading 正道 | linecache lc_scan 现状 |
|--------|------------------|----------------------|
| 空树加载 | 叶满方裂，填充率 ~100% | 裂叶取中点 (50%/50%)，填充率 ~50% |
| 非空树追加 | 逐条插入(本该另法) | 逐条插入：`lcB_fitleaf` → `lcB_makeroom` |
| 分裂策略 | 仅最右路径 | `lcB_makeroom` 自上向下逐层预裂 |
| 批量建树 | 自底向上：先填满叶 → 叶推内节点 → 内节点推根 | 先逐条收集到 buf 再建树 (旧)/ pend 式 (新 lcB_scan_empty) |
| 新 `lcB_scan_empty` | 正确：叶满填，pend 式自底向上 | 仅空树用；非空树仍逐条插入 |

linecache 的 `lc_scan` 分两路：
- **空树路径** (`lcB_scan_empty`)：叶满填 + pend 式自底向上建树，正确。
- **非空树路径**：游标追加，逐条调 `lcB_fitleaf` → `lcB_makeroom` 裂叶取中点，填充率仅 ~50%。

混合策略的根源：空树创建新叶可任意填满；非空树须在已有叶中插入，分裂取中点不可免。

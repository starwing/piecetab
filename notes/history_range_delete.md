# 计量B+树区间删除算法演进与终极候选设计论

## 一、 算法演进历史之辩 (Evolution of Algorithms)

计量 B+ 树（Metric B+ Tree）进行区间批量删除（Range Delete / Splice），其核心难点在于既要高效率（$O(\log N)$）大块吞噬，又要维持 B+ 树严格的半满（Underfull）约束与账目精确。

本项目历经数代算法演进，其得失如下：

### 1. 第一代：步进法 (Step-by-Step Sweeping)
*   **心智模型**：以“待删字节 `del`”为主线。自游标 $L$ 起，沿叶向右蚕食。若叶尽而余量犹存，则沿物理路径上溯求后驱，再下行削之。
*   **败因**：边删除数据、边调整树形。一旦触发节点合并或缩根，树形拓扑变幻，度量（Metrics）大乱。游标所存路径（`paths`）随指针重组而瞬间崩坏，导致后续步骤指向废墟。

### 2. 第二代：双游标拉链法 (LR Zipper - First Version)
*   **心智模型**：预先扣减待删量。拷贝游标 $L$ 推进 `del` 长度得 $R$。自底向上，裁剪两端边缘，剪除中段废弃子树，最后缝合 $L$ 与 $R$。
*   **败因**：若底层节点因裁剪而 underfull，而在分歧层之下又无同父兄弟可借，则自底向上的单次合并注定留下无法填平之底坑。

### 3. 第三代：三段法 (Three-Phase Repair - Current Implementation)
*   **心智模型**：自底向上剪枝（Prune & Trim），允许边缘残缺；自顶向下输血（Proactive Top-Down），带着上层富余资源沿崖壁向下借位、合并；自底向上重平衡（Bottom-Up Rebalance），修补宗主不足。
*   **败因**：在第一段或第二段中，若 $L$ 与 $R$ 对应层发生物理合并（Merge），则 $R$ 游标之路径彻底失效。更核心地，合并将原有裂缝“包裹”入单一节点内，在更深层催生了**两条全新的、需要修补的局部裂缝路径**。

### 4. 第四代：显式分歧点栈/队列驱动 (Explicit Stack/Queue)
*   **心智模型**：在 `splicerange` 栈帧中声明一个大小为 `LC_MAX_LEVEL = 16` 的局部数组作为工作栈。Up 阶段真合并产生的新分歧点入栈；之后自顶向下 pop 解决。
*   **优点**：扁平化、解耦极佳，零动态内存分配。
*   **缺点**：需显式声明并维护一个约 160 字节的局部栈结构。

### 5. 第五代：R->paths 隐式栈与紧邻指针重写方案 (R->paths Implicit Stack & Downward Fixup - Ultimate Candidate)
*   **心智模型**：直接复用 `R->paths` 作为天然的合并状态栈，在 top-down 发生槽位变动时，实时顺手重写下一层 `R->paths[dl+1]` 指针。
*   **优点**：**极致完美的零空间、零分配、高内聚方案。**

## 二、 症结痛点第一性原理解析 (The Root Cause)

凡 B+ 树区间删除之失败，其根源既非指针不稳，亦非度量错算，乃是：
$$\text{合并操作的级联效应（Cascading Merges）在深层催生了呈线性增长的“新相邻分歧点”。}$$

设在 $dl$ 层，左崖 $Lr$ 与右崖 $Rl$ 满足合并条件（$cl + cr \le FANOUT$）：
合并后，$Rl$ 被销毁，原 $Lr$ 最右子节点 $C_{left}$ 与原 $Rl$ 最左子节点 $C_{right}$ 在 $dl+1$ 层**骤然相邻**。此二子节点极大概率皆处于 underfull 状态，这便在 $dl+1$ 层产生了一个**全新的、原先不存在的分歧点** $(Lr, \text{idx}(C_{left}), \text{idx}(C_{right}))$。

**此乃单向 up+down 架构之死穴**：Phase Down 沿着一条既定轨道下行，而合并却在轨道旁侧不断开辟新的“支线战场”。

## 三、 终极候选方案精细设计与论证

针对合并产生的深层相邻分歧问题，我们将“显式工作栈方案”与尊驾提议的“R->paths 隐式栈与紧邻重写方案”作并列推演与详细论证。

### 1. 显式分歧点栈方案 (Explicit Stack)

在 `splicerange` 局部声明 `LC_MAX_LEVEL` 深度工作栈。

```c
typedef struct {
    lc_Node *parent;   /* 父节点 */
    short    ileft;    /* 左孩子在 parent->children 中的索引 */
    short    iright;   /* 右孩子在 parent->children 中的索引 */
} lc_Divergence;
```

*   **Up 阶段**：裁剪并真合并。若 $dl$ 层合并，将下一层新相邻分歧点 `(nl, mLR - 1, mLR)` 入栈。
*   **Down 阶段**：
    ```c
    while (stack_top > 0) {
        lc_Divergence div = stack[--stack_top];
        /* 在 parent 之下，对 ileft 和 iright 进行合并或平分 */
        /* 若合并，再次将新产生的下层分歧点入栈 */
    }
    ```

### 2. 尊驾所提之“R->paths 隐式栈与紧邻指针重写方案” (Ultimate)

#### 核心突破：以“R->paths[dl]”作状态锁，并在 $dl$ 操作时自愈 $dl+1$ 游标。

#### A. 算法核心步骤：
1.  **Bottom-Up 阶段**：自底向上剪枝。
    *   在 $dl$ 层，若 $L_{dl}$ 与 $R_{dl}$ 能够合并，行真合并，并将 `R->paths[dl]` 置为 `NULL`。
    *   若超载无法合并，保留 `R->paths[dl]`（指向右裁剪节点的原始位置）。
2.  **Top-Down 阶段**：自顶向下处理。
    *   **若 `R->paths[dl] == NULL`**：代表在此层或上层已合并。此时不需要平分。我们只需要对当前唯一的左路径节点进行严格的半满（$\ge FANOUT/2$）校验，若不足，行标准的单点 rebalance（向左借或合并）。
    *   **若 `R->paths[dl] != NULL`**：代表未合并，必须执行 `Redistribute` 平分。
3.  **寻找 $R_{dl}$ 父节点（无脑向上溯源）**：
    *   若 `R->paths[dl-1] != NULL`：说明父节点在上一层未被合并，其 parent 即为 `*R->paths[dl-1]`。
    *   若 `R->paths[dl-1] == NULL`：说明父节点在上一层已被合并。此时 $R_{dl}$ 的父节点和 $L_{dl}$ 相同，皆归于 `*L->paths[dl-1]`。
4.  **实时顺手重写下一层 `R->paths[dl+1]` 指针（消除张冠李戴）**：
    在 $dl$ 层执行平分或合并时，因为发生了子节点数据挪移，下一级指针 `R->paths[dl+1]` 对应的槽位必须被同步更新：
    *   *以平分为例*：若平分致使原属于 $R_{dl}$ 的前 $d$ 个孩子移入了 $L_{dl}$ 的尾部（原 $L_{dl}$ 拥有 $cl$ 个孩子），则：
        *   原属于 $R_{dl}$ 的第 0 个孩子（即 $dl+1$ 层指向的相邻点），其新物理槽位变为了 `&L_dl->children[cl]`。
        *   我们直接在一行内更新游标指针：
            `R->paths[dl+1] = &L_dl->children[cl];`

#### B. 拓扑安全性数学证明 (No Cascading Breakdown)
*   **引理**：在 $dl$ 层挪移 $dl+1$ 层的子节点时，**只需修正下一层游标 `R->paths[dl+1]`**，更深层级（$dl+2 \dots levels$）均无需修正。
*   **证明**：
    设节点 $C$（属于 $dl+1$ 层）从右父节点 $R_{dl}$ 转移到了左父节点 $L_{dl}$。
    虽然 $C$ 在父节点 `children` 数组中的物理槽位发生了变动，但 **$C$ 节点本身的内存地址没有发生改变**。
    在 $dl+2$ 层，`R->paths[dl+2]` 指向的是 `&C->children[idx]`。
    因为 $C$ 本身内存未变，其内部的 `children` 数组亦未发生重组，故 `&C->children[idx]` 的物理内存槽位地址**分毫不差地保持静态合法**。
    因此，深层游标天然安全，无级联崩溃之险。证毕。

#### C. 会计账目（Metrics Update）极致简化
*   若 `R->paths[dl-1] == NULL`（已合归同父）：
    平分/合并带来的度量转移，仅在共同的父节点 `L_dl-1` 内部，在 `bytes[il]` 和 `bytes[ir]` 之间进行转移。
    **对父节点之上的所有祖先度量和没有任何影响**（相互抵消）。无需向两边传播，不产生任何会计误差。
*   若 `R->paths[dl-1] != NULL`（未合归）：
    调用 `lcM_tx(L, R, dl-1, db, dl)` 向各自路径的祖先分道累加/累减。

## 四、 终极方案对决 (Candidates Tradeoffs)

| 维度 | 函数递归方案 (Recursive) | 显式栈方案 (Explicit Stack) | R->paths 隐式栈与重写方案 (R->paths Hack) |
| :--- | :--- | :--- | :--- |
| **空间开销** | 依赖函数调用栈（$O(H)$ 级栈帧） | 显式局部栈（约 160 字节，在栈帧内） | **0 额外空间（纯原位复用）** |
| **指针安全性** | 极其脆弱，递归返回易踩空 | **极高**（不存二级槽位，存稳定节点指针与相对索引） | **极高**（通过在 $dl$ 顺手重写 $dl+1$ 槽位，保持绝对正确） |
| **会计账目更新** | 后序求和，账目最准 | 差量传播 + 局部重算 | **极致精简**（父层为空则局部对调，父层不空则 lcM_tx 传播） |
| **代码量与复杂度** | 约 300+ 行，极其晦涩 | 约 200 行，多出栈管理逻辑 | **约 150 行以内，极其精干利落** |

## 五、 重构指南

本项研究最终确定：**第五代：R->paths 隐式栈与紧邻指针重写方案**，为计量 B+ 树区间删除之终极解决方案。

重构时应重点保障：
1.  **Bottom-Up 真合并断链**：合并时 `R->paths[dl] = NULL`。
2.  **Top-Down 动态重组槽位**：在执行 `Redistribute` 或 `Merge` 时，凡发生数据移动，必须立刻修正下级 `R->paths[dl+1]` 指向挪移后的新槽位。
3.  **度量更新**：严密比对上一层 `paths[dl-1]` 是否为空，空则局部对消，不空则双路传播。
